I'm a senior-year student at the University of British Columbia, where I've spent many years changing paths. I initially majored in biology, switched to mathematics, and then finally settled on computer science - a diverse field of study with a wonderful blend of theory and applications. I spent the first half of my degree fascinated by the theoretical side of math and computer science, but the ideas eventually became too abstract (and confusing) for my tastes. I started to want something more applied in my education besides my usual routine of taking notes and doing practice problems.

A major turning point in my interests then came during the 2020-2021 academic year, which was a new experience in multiple ways. School was moved entirely online due to the COVID-19 pandemic, and my coursework placed a heavy emphasis on large programming projects instead of conventional exams. Although I already learned how to program using basic data structures and algorithms in previous CS courses, it wasn't until the 2020-2021 academic year that I started to get a sense of all the joys and complexities of software engineering. My education that year really could be described as "learning by building things". This was the idea that the best way to understand a new concept was to incorporate it into a programming project. In doing so, having to think about things like edge cases, error handling and performance optimizations naturally led to a deeper understanding that I would never have been able to achieve just by passively reading notes. Over the course of that year, I went from knowing little of web development to having an appreciation for the details of things like asynchronous Javascript, SQL query optimization and the TCP protocol - simply because understanding these details were crucial for my projects to work. Above all, this active style of learning was a lot of fun.

While these projects pushed me to improve my CS fundamentals, the most valuable experience that I gained from them was really the process of learning to work with other programmers. Since these projects were much more complicated than any programming assignment I had done before, I was required to work with a fixed team of other students for the entire term in order to finish on time. Essentially, here's how many of my classes went that year:

  1. Get assigned to a particular project with teammates I've never met before at the start of term
  2. Become intimidated by the required technologies and the project specification
  3. Panic!
  4. Learn as much as I can from the web and my teammates
  5. Navigate through countless setbacks and successes with my group over many weeks
  6. Repeat step 5 for the rest of the term.
  7. Submit the project, with a deeper understanding of myself as a learner and teammate

As part of taking CPSC 310, my university's software engineering course, I was assigned to work on this web development project with a partner for the entire semester. Both the project specification and the tech stack were predetermined by the course staff, which meant that we had to teach ourselves how to use new languages and frameworks. The amount of provided starter code was minimal - my partner and I were responsible for designing, implementing and testing a solution that fulfilled the project requirements. The project goal was to create a Typescript program that allows the user to query for information regarding courses and rooms at UBC. Each sprint emphasized a particular aspect of software development such as unit testing, server-side asynchronous logic and frontend data validation. In more detail, here's how the project went:

Sprint 0: Following the principles of test-driven development, I reinforced my understanding of the project specification by writing a set of over 60 black-box unit tests for functionality that was yet to be implemented. Since no sample solution was provided for any part of the project, this test suite was the only means of verifying the correctness of my work, aside from submitting directly to the autograder and risking a grade deduction if any regression tests failed. Trying to achieve a comprehensive yet parsimonious test suite forced me to not only think carefully about the program's expected behaviour for all possible inputs, but also ensure that my test data were actually representative of the use cases implied in the project specification.

Sprint 1: This sprint was all about backend development, which was roughly split into two parts. In the first part, my partner and I developed a mechanism for loading UBC course information from an input zip file into our custom-designed data model on the local disk. The second part's objective was to enable the backend to respond to queries about the data, which were submitted by the user in a custom syntax. It was surprisingly tricky to correctly automate the process of checking the syntax and semantics of the queries, in addition to getting the backend to return the correct data. Ultimately, these challenges allowed me to become more fluent in Javascript - the large amount of file I/O involved in this sprint, for example, meant that I quickly became used to working with promises and callbacks.
